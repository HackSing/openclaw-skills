# openclaw-skills 维护说明

版本：2026-03-07

## 目标

这个仓库用于和 OpenClaw 本机运行目录的 `~/.openclaw/skills` 保持长期双向维护。

原则：

- 运行目录是可执行真相
- Git 仓库是版本管理与跨端同步基线
- 两端都可能产生有效更新
- 同步时必须排除运行缓存和临时备份文件

---

## 双向同步原则

### 1. 本地运行目录改了

适用场景：

- 在本机直接优化了 skill
- 调整了 `SKILL.md`
- 修改了脚本、配置模板或辅助代码

处理方式：

1. 对比 `~/.openclaw/skills` 和仓库
2. 将本地有效变更同步到仓库工作副本
3. 检查差异是否符合预期
4. commit
5. push

### 2. 其他端先改了仓库

适用场景：

- 在另一台机器上更新了某个 skill
- 仓库里新增了 examples、resources、scripts、data 等辅助资料

处理方式：

1. 拉取仓库最新版本
2. 对比仓库和本地运行目录
3. 将仓库中的有效更新同步回本地运行目录
4. 保持本地运行目录可直接使用

### 3. 两端都有变化

适用场景：

- 本地运行目录有新修改
- 仓库远程也有其他端提交的新内容

处理方式：

1. 先拉仓库最新版本
2. 做双向对比
3. 本地运行目录中的主技能改动同步到仓库
4. 仓库中的新增资料和其他端优化同步回本地
5. 验证两边收敛后再 commit 和 push

---

## 目录边界

### 应该同步的内容

- `SKILL.md`
- skill 自带脚本
- examples
- resources
- scripts
- data
- 其他明确属于 skill 维护资产的文件

### 不应该同步的内容

- `__pycache__/`
- `*.bak.*`
- 本地临时调试产物
- 一次性日志文件
- 不属于 skill 的系统缓存

---

## 推荐流程

### 日常同步流程

```bash
1. 拉取仓库最新版本
2. 对比本地 skills 和仓库
3. 双向同步有效文件
4. 清理缓存和备份文件
5. 验证 diff 收敛
6. commit
7. push
```

### 验证要求

同步后至少确认：

- 目标 skill 在本地运行目录可正常读取
- 仓库工作区变更符合预期
- 不存在误同步的缓存文件
- 双方目录在忽略缓存后已收敛

---

## 当前维护策略

当前仓库采用：

- **方案 B**
- 即：以本地运行目录和仓库双向维护为准
- 仓库中的额外资料目录需要同步回本地
- 本地运行目录中的最新可执行修改也需要同步进仓库

这意味着本仓库不是单向镜像仓，而是一个和运行目录协同演进的 skills 基线仓库。
