# 你的 AI Agent 还在「被动等指令」？是时候让它自己上班了

> 定时任务 + 心跳，两个机制让你的 Agent 从「工具」进化成「同事」。

---

你有没有遇到过这样的场景——

每天早上打开电脑，手动跑一遍数据采集脚本。然后打开 Agent，告诉它"帮我看看昨天有没有遗漏"。再手动检查一遍它的记忆文件，确认没写错东西。

一天两天还行，一周下来你发现：**你花在"管理 Agent"上的时间，比 Agent 替你省下的时间还多。**

问题出在哪？

你的 Agent 缺少两样东西：**一个闹钟，和一面镜子。**

闹钟让它知道什么时候该干活——这就是**定时任务**。
镜子让它定期审视自己的状态——这就是**心跳**。

OpenClaw 把这两个机制做成了 Agent 工作空间的标配。今天我们从原理到实战，把它讲透。

---

## 一、定时任务：给 Agent 装上闹钟

### 它解决什么问题？

你希望 Agent 每隔 30 分钟去刷一遍社交媒体、每天早上 9 点自动生成日报、每周五下午跑一次数据备份——这些"周期性、无需人工触发"的工作，就是定时任务的领地。

**一句话定义：定时任务是面向外部世界的周期性自动化执行机制。**

它关注的是"做事"——采集数据、处理信息、推送通知。

### 核心机制

定时任务在 OpenClaw 中有两种实现路径：

**路径一：Skill 级别的轮询调度**

以 `social-reader` 这个 Skill 为例，它通过 `sources.json` 配置文件定义巡逻计划：

```json
{
    "accounts": [
        { "username": "dotey", "label": "宝玉-AI资讯+实战" },
        { "username": "op7418", "label": "歸藏-生图专家" },
        { "username": "vista8", "label": "向阳-AI趋势" }
    ],
    "fetch_interval_minutes": 30,
    "max_tweets_per_account": 5
}
```

Skill 内部实现了一条完整的 Pipeline：

```
Watcher（抓取） → Processor（LLM 蒸馏） → Action（通知/审阅）
```

每个节点各司其职：

| 节点 | 脚本 | 职责 |
|------|------|------|
| **Watcher** | `watcher.py` | 读取 sources.json，抓取新帖子，通过 seen_ids.json 去重，写入 pending_tweets.json |
| **Processor** | `processor.py` | 读取待处理队列，调用 LLM 生成评论/摘要，输出到 drafts.json |
| **Action** | `notifier.py` | 启动本地 HTTP 审阅服务（端口 18923），提供批准/拒绝/改写/归档操作界面 |

**路径二：Claude Code 内置定时任务**

Claude Code 原生支持基于 cron 表达式的定时调度。你可以直接创建一个定时任务，让 Claude 在指定时间自动启动一个会话来执行 Skill：

```
任务 ID：social-patrol
调度表达式：0 9,13,18 * * 1-5  （工作日的 9:00、13:00、18:00）
执行内容：运行 social-reader Pipeline，收集并蒸馏新内容
```

cron 表达式使用本地时区，格式为：`分 时 日 月 周几`

| 表达式 | 含义 |
|--------|------|
| `0 9 * * *` | 每天早上 9:00 |
| `0 9 * * 1-5` | 工作日早上 9:00 |
| `30 8 * * 1` | 每周一早上 8:30 |
| `0 0 1 * *` | 每月 1 号午夜 |

除了 cron 周期调度，还支持**一次性定时触发**，使用 ISO 8601 时间戳，比如 `2026-03-15T14:30:00+08:00` 表示在 3 月 15 日下午 2:30 执行一次后自动停用。

### 关键字段一览

**Skill 配置层（sources.json）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `accounts` | Array | 监控的社交媒体账号列表 |
| `accounts[].username` | String | 账号用户名 |
| `accounts[].label` | String | 人类可读的标签描述 |
| `fetch_interval_minutes` | Integer | 轮询间隔（分钟），如 30 表示每 30 分钟巡逻一次 |
| `max_tweets_per_account` | Integer | 每个账号单次最多抓取的帖子数 |

**Claude Code 调度层：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `taskId` | String | 任务唯一标识，kebab-case 格式（如 `daily-digest`） |
| `prompt` | String | Agent 每次执行时的完整指令 |
| `description` | String | 任务的一行简述 |
| `cronExpression` | String | 5 位 cron 表达式（周期性任务） |
| `fireAt` | String | ISO 8601 时间戳（一次性任务），与 cron 互斥 |
| `enabled` | Boolean | 是否启用自动执行 |

### 状态管理

定时任务依赖一组文件来维护运行状态：

| 文件 | 用途 |
|------|------|
| `data/seen_ids.json` | 去重缓存，记录已处理的帖子 ID |
| `data/pending_tweets.json` | 待处理队列 |
| `data/drafts.json` | LLM 蒸馏后的草稿 |
| `data/archive.json` | 历史归档记录 |

去重机制确保同一条内容不会被反复处理——Watcher 每次运行时都会检查 `seen_ids.json`，只有新内容才会进入 Pipeline。

## 二、心跳：给 Agent 装上一面镜子

### 它解决什么问题？

你的 Agent 昨天犯了一个错误，它今天还记得吗？它上周学到一个有价值的经验，这个经验被沉淀下来了吗？它的记忆文件是不是越来越臃肿，需要精炼了？

如果没人提醒，Agent 会"活在当下"——每次对话都是全新的开始，过去的经验随风而逝。

**心跳就是解决这个问题的。**

**一句话定义：心跳是面向 Agent 自身的周期性自我反思与维护机制。**

它关注的不是"做事"，而是"审视自己"。

### 核心机制：四项检查闭环

心跳的设计哲学是**最小可跑通闭环**——不搞复杂，只做最核心的四件事：

```
┌──────────────────────────────────────┐
│           HEARTBEAT 触发             │
├──────────────────────────────────────┤
│                                      │
│  ① 错误追踪                          │
│     今天是否有新的错误或被纠正？        │
│                                      │
│  ② 学习池                            │
│     是否需要写入 .learnings/？         │
│                                      │
│  ③ 记忆蒸馏                          │
│     是否需要把经验提炼进 MEMORY.md？    │
│                                      │
│  ④ 外部学习候选                       │
│     是否有需要用户确认的外部学习？       │
│                                      │
├──────────────────────────────────────┤
│  无异常 → HEARTBEAT_OK               │
│  有发现 → 输出具体报告                 │
└──────────────────────────────────────┘
```

逐项拆解：

**检查一：错误追踪**

回顾最近的交互记录，检查是否出现过新的错误、异常行为、或者被用户纠正的情况。如果有，记录下来作为学习素材。

**检查二：学习池**

判断是否有值得沉淀的经验（比如一个反复出现的调试技巧、一个容易踩的坑）。如果有，写入 `.learnings/` 目录。

**检查三：记忆蒸馏**

检查 `.learnings/` 和 daily memory（`memory/YYYY-MM-DD.md`）中是否有高价值内容需要"提拔"到长期记忆 `MEMORY.md` 中。这就像人类把短期记忆中重要的部分固化成长期记忆。

**检查四：外部学习候选**

有些外部来源的知识（比如从网页、文档中获取的信息）需要用户确认后才能纳入学习体系。心跳会标记出这些候选项，等待用户拍板。

### 运行规则

心跳机制遵循严格的设计原则：

| 原则 | 说明 |
|------|------|
| **最小可跑通** | 不要把心跳写得太复杂，能跑通四项检查即可 |
| **结构化** | 输出格式清晰，便于追踪和回溯 |
| **可执行** | 每项检查都有明确的 action item，不是空谈 |
| **留有扩展空间** | 四项是核心最小集，后续可按需添加新检查项 |
| **可以沉默** | 如果一切正常，直接回复 `HEARTBEAT_OK`，不制造噪音 |

### 多 Agent 场景的关键规则

这是一个容易踩的坑：**在多 Agent 协作场景下，每个 Agent 读取的是自己工作空间根目录下的 `HEARTBEAT.md`，而不是共享同一份文件。**

```
workspace-agent-A/
├── HEARTBEAT.md          ← Agent A 读这份
├── MEMORY.md
└── .learnings/

workspace-agent-B/
├── HEARTBEAT.md          ← Agent B 读这份
├── MEMORY.md
└── .learnings/
```

每个 Agent 有自己独立的反思空间。如果让所有 Agent 共用一个 HEARTBEAT.md，它们的错误记录和学习经验就会混在一起，完全无法追溯。

### 心跳在工作空间五层模型中的位置

OpenClaw 的工作空间采用五层架构：

```
① 身份层 Identity    →  SOUL.md / IDENTITY.md / USER.md
② 运维层 Operations  →  AGENTS.md / HEARTBEAT.md / TOOLS.md / NOTICE.md
③ 知识层 Knowledge   →  MEMORY.md / memory/YYYY-MM-DD.md
④ 上下文层 Context   →  context/ / shared-context/
⑤ 学习层 Learning    →  .learnings/ / reviews/
```

心跳位于**运维层**，它是 Agent 自我管理的核心协议。它向上读取身份信息（我是谁），向下驱动知识层和学习层的更新（该记住什么、该忘记什么）。

## 三、定时任务 vs 心跳：别搞混了

初看之下，定时任务和心跳都是"周期性执行某些事情"，很容易混淆。但它们的设计意图完全不同：

| 维度 | 定时任务 | 心跳 |
|------|---------|------|
| **面向对象** | 外部世界（数据、用户、服务） | Agent 自身 |
| **核心目的** | 自动化执行业务逻辑 | 自我反思与维护 |
| **触发方式** | 时间间隔 / cron 表达式 / 一次性定时 | 周期性（通常每次会话或每天） |
| **输出产物** | 处理结果（草稿、通知、数据） | 学习记录、记忆更新、`HEARTBEAT_OK` |
| **状态管理** | 队列 + 缓存文件（seen_ids、pending、drafts） | 文件系统（.learnings/、MEMORY.md） |
| **配置方式** | JSON 配置 + cron 表达式 | HEARTBEAT.md 文件 |
| **多 Agent 场景** | 可共享数据源（如 sources.json） | 严格隔离（每个 Agent 独立 HEARTBEAT.md） |
| **类比** | 闹钟 + 任务清单 | 日记 + 自我复盘 |

**打个比方：**

定时任务是你给员工排的值班表——几点干什么活，干完把结果交上来。
心跳是员工自己的工作日志——今天犯了什么错、学到了什么、有什么需要汇报的。

一个对外，一个对内。缺一不可。

## 四、实战案例

光说不练假把式。下面用几个真实场景来演示这两个机制怎么落地。

### 案例一：社交媒体自动巡逻站

**需求：** 每 30 分钟自动监控 7 个 AI 领域 KOL 的推文，经过 LLM 蒸馏后推送到审阅界面。

**第一步：配置数据源**

编辑 `social-reader/sources.json`：

```json
{
    "accounts": [
        { "username": "dotey", "label": "宝玉-AI资讯+实战" },
        { "username": "oran_ge", "label": "橘子-AI创业认知" },
        { "username": "op7418", "label": "歸藏-生图专家" },
        { "username": "vista8", "label": "向阳-AI趋势" },
        { "username": "xiaohu", "label": "互总-AI资讯" },
        { "username": "yangyi", "label": "曾-AI落地" },
        { "username": "lxfater", "label": "铁锤人-技术+营销" }
    ],
    "fetch_interval_minutes": 30,
    "max_tweets_per_account": 5
}
```

**第二步：创建定时任务**

```
任务 ID：social-patrol
cron 表达式：*/30 9-22 * * *
含义：每天 9:00 到 22:00，每 30 分钟执行一次
执行指令：运行 social-reader Pipeline 全流程
```

**第三步：运行**

```bash
# 全 Pipeline 一键运行
python scripts/run_pipeline.py

# 也可以只跑某个节点
python scripts/run_pipeline.py --watch-only    # 只抓取
python scripts/run_pipeline.py --process-only  # 只蒸馏
python scripts/run_pipeline.py --notify-only   # 只通知
```

**效果：** 你不用盯着 Twitter 刷屏了。每半小时，系统自动抓取 → 去重 → LLM 蒸馏 → 推送到审阅页面，你只需要在审阅页面点"批准"或"改写"就行。

### 案例二：Agent 每日健康自检

**需求：** 让 Agent 每天早上对自己进行一次全面自检，确保记忆系统健康运转。

**心跳执行流程：**

```
Agent 启动 → 读取 HEARTBEAT.md → 执行四项检查 → 输出报告
```

**场景模拟：**

Agent 在进行心跳检查后发现：

```
HEARTBEAT 检查报告：

① 错误追踪：
   - 昨天在处理 allstock-data 请求时，错误地使用了过期的 API 端点
   - 已记录到 .learnings/2026-03-09-api-endpoint-fix.md

② 学习池：
   - 新增学习：social-reader 的 fetcher.py 支持长文章类型（type: "article"），
     之前误以为只支持推文
   - 已写入 .learnings/2026-03-09-article-type.md

③ 记忆蒸馏：
   - .learnings/ 中有 3 条关于 API 调用的经验可以合并精炼
   - 建议提炼为 MEMORY.md 中的 "API 调用注意事项" 条目

④ 外部学习候选：
   - 无需用户确认的新内容

状态：发现 3 项需处理，已自动完成 2 项，1 项待用户确认（记忆蒸馏）
```

如果一切正常，输出就简单得多：

```
HEARTBEAT_OK
```

**价值：** Agent 不再是"金鱼记忆"。它会持续积累经验、修正错误、精炼知识，越用越聪明。

### 案例三：定时巡逻 + 心跳联动

**需求：** 一个完整的内容运营 Agent，既能自动采集内容，又能持续优化自己的处理质量。

**架构设计：**

```
     ┌─────────────────────────────────────────┐
     │            每 30 分钟触发                  │
     │                                         │
     │   定时任务                                │
     │   ├── Watcher: 抓取 KOL 新推文            │
     │   ├── Processor: LLM 蒸馏评论             │
     │   └── Action: 推送审阅通知                 │
     │                                         │
     ├─────────────────────────────────────────┤
     │            每天早上 9:00 触发              │
     │                                         │
     │   心跳                                   │
     │   ├── 检查：昨天蒸馏的内容质量如何？         │
     │   ├── 检查：用户拒绝了哪些草稿？为什么？     │
     │   ├── 学习：把"高通过率"的写法沉淀下来       │
     │   └── 优化：更新 prompt.txt 中的写作规则    │
     │                                         │
     └─────────────────────────────────────────┘
```

**关键点：心跳驱动定时任务的质量提升。**

比如 Agent 通过心跳发现：最近一周用户拒绝率最高的草稿，都是那些"只有摘要没有观点"的内容。于是它把这个发现写入 `.learnings/`，并在下次蒸馏时加重"输出洞察而非摘要"的权重。

这就形成了一个**自我进化的闭环**：

```
定时任务产出内容 → 用户反馈 → 心跳复盘 → 沉淀经验 → 优化定时任务的执行质量
```

### 案例四：一次性提醒 + 周期任务组合

**需求：** 下周三下午 3 点有一个产品发布会，需要提前准备内容素材，发布后持续追踪反馈。

**阶段一：一次性定时任务（发布前）**

```
任务 ID：launch-prep
触发时间：2026-03-18T09:00:00+08:00（发布会前 6 小时）
执行内容：
  1. 运行 social-reader 抓取竞品最新动态
  2. 用 social-writer 生成 3 条预热推文草稿
  3. 推送审阅通知
```

这个任务用 `fireAt` 一次性触发，执行完自动停用。

**阶段二：周期定时任务（发布后）**

```
任务 ID：launch-monitor
cron 表达式：0 */2 * * *（每 2 小时一次）
执行内容：
  1. 抓取社交媒体上关于本次发布的讨论
  2. LLM 分析舆情倾向
  3. 有负面反馈时立即通知
```

**阶段三：心跳收尾**

发布一周后，心跳在日常检查中发现：

```
② 学习池：
   - 本次发布追踪中发现，用户对"技术细节类"推文的互动率
     是"功能介绍类"的 3 倍
   - 建议写入长期记忆作为内容策略参考
```

**三个阶段无缝衔接**：一次性任务处理突发事件，周期任务持续追踪，心跳负责经验沉淀。

## 五、快速上手指南

如果你现在就想用起来，只需要三步：

**Step 1：配置你的第一个定时任务**

从 `social-reader` 开始，编辑 `sources.json`，填入你想监控的账号和巡逻间隔。

**Step 2：跑通 Pipeline**

```bash
python scripts/run_pipeline.py
```

确认 Watcher → Processor → Action 三个节点都能正常执行。

**Step 3：启用心跳**

在你的 Agent 工作空间根目录创建 `HEARTBEAT.md`，写入四项检查的模板。让 Agent 在每次会话开始时执行一次心跳检查。

---

## 写在最后

定时任务和心跳，一个管"做事"，一个管"反思"。

这恰好对应了高效工作的两个核心要素：**执行力**和**复盘能力**。

单有定时任务，Agent 是一台不会累的机器，但永远不会进步。
单有心跳，Agent 是一个善于反思的哲学家，但什么正事也不干。

两者结合，才是一个**既能干活又能成长的 AI 同事**。

OpenClaw 的设计理念很简单：别把 Agent 当工具使，把它当队友培养。
